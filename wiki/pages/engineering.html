Engineering Overview

One platform, three coordinated layers of control.

The engineering goal of Metlock is not only to recover zinc, but to do so in a way that stays tunable, measurable, and extensible. The project is organized so that each module solves a different engineering problem: controlled zinc import, predictable gene-expression calibration, and zinc binding through dedicated protein systems.

That structure matters because each module reduces uncertainty for the others. Calibration helps define safe expression windows, import work tests functional uptake under induction, and binding work expands the platform beyond one intracellular strategy. Taken together, the modules create an engineering workflow that is easier to interpret, easier to compare, and easier to improve over time.

Engineering focus Metlock is being built as a system that can be tuned deliberately, not as a one-off construct that only works under a single condition.
Design-Build-Test-Learn Framework

How the project moves through iteration.

The design phase begins by deciding which parts of zinc handling should be controllable and which measurements are needed to evaluate success. This leads to a platform that emphasizes induction control, shared assay logic, and clearer comparison between modules rather than isolated experimental snapshots.

The build phase translates those decisions into constructs and workflows that can be tested under comparable conditions. The test phase then asks whether the platform behaves as expected across growth, viability, uptake, calibration, or binding readouts. The learn phase uses those results to narrow useful operating windows and guide the next round of design.

Module Integration

How the modules support one another.

Module 1 establishes controllable zinc import. Module 2 defines how the shared control backbone behaves before the project relies on functional proteins. Module 3 broadens the system by comparing zinc-binding strategies outside and inside the cell. Each module answers a different engineering question, but all three are meant to share logic rather than compete for attention.

That integration is what makes the platform useful. Import is easier to interpret when calibration already defines reasonable expression windows, and binding work becomes easier to compare when the rest of the project already uses a consistent measurement framework.

Key Technical Decisions

Why the architecture is organized this way.

One major decision is the reuse of a shared induction language across modules. This creates a more stable basis for comparison and reduces the risk that each branch of the project becomes difficult to interpret against the others.

Another major decision is to calibrate expression before relying fully on functional metal-handling proteins. That keeps the project grounded in measurable, lower-risk expression behavior before moving into designs that may create stronger burdens on the cells.

A third decision is to treat controls as part of the engineering logic itself. Growth, viability, depletion, retention, and comparison controls are not separate from the design process; they are part of how the team decides whether a construct is actually useful.

Decision logic The engineering story is strongest when design choices can be traced clearly to why they reduce uncertainty or improve comparison.
