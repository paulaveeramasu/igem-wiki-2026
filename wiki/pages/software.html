Software and Dry Lab

Using computation to make Metlock easier to interpret.

The software side of Metlock is meant to turn experimental ideas into something we can compare, tune, and learn from more systematically. In plain English, the dry-lab work helps us understand how different induction choices and promoter strengths might affect expression before future wet-lab data becomes difficult to interpret.

This page explains what the computational workflow is for, what kinds of inputs it uses, what outputs it can generate, and how it supports Module 2 as a calibration layer for the rest of the project.

Software Overview

What this page is for.

The software work behind Metlock is designed to support reasoning, not replace experiment. Its purpose is to help the team organize assumptions, simulate expected expression behavior, compare promoter choices, and turn raw or future fluorescence data into something that can be interpreted more clearly.

At this stage, the page is best understood as a computational companion to the rest of the platform. It provides a place to explain how dry-lab methods support design choices, especially around AHL dose response, promoter strength, expression bands, and how those choices may affect later experimental comparisons.

Dry-lab role The software side of the project helps define expectations, normalize interpretation, and make future data easier to compare across conditions.
Workflow

A simple computational flow.

The workflow begins with a defined set of inputs, such as promoter identity, AHL dosing conditions, coding-sequence context, time-course measurements, or future fluorescence and growth data. From there, the software can help normalize values, compare trajectories, estimate expression behavior, and produce outputs that are easier to interpret visually.

  1. Input data Promoter choice, AHL conditions, fluorescence measurements, growth measurements, or curated reference assumptions enter the pipeline.
  2. Processing and normalization Data can be organized into comparable forms such as RFU/OD, cleaned time courses, or grouped conditions for later analysis.
  3. Analysis The software can examine dose-response behavior, promoter-linked expression differences, timing patterns, or candidate expression bands.
  4. Outputs and visualization The final products can include plots, summary tables, ranked comparisons, and reusable figures that help guide future experiments.
Connection to Module 2

How the software supports sfGFP calibration.

The clearest connection between software and the rest of the project is Module 2. That module uses sfGFP as a calibration layer so the team can understand how strongly the shared control backbone turns on before moving into more functionally demanding proteins.

Computationally, that means the software can help compare AHL dose-response trends, map conditions against Anderson promoter strength, and identify expression bands that appear strong enough to be useful without immediately suggesting excessive burden. In that role, the dry-lab work acts as an interpretation layer for the calibration experiments rather than a separate project.

Module 2 connection Software gives the team a way to translate fluorescence and growth observations into a clearer expression map for later design decisions.

Future Use

How later team members can reuse this page.

This page should stay useful even as the project changes. Future team members should be able to plug in new fluorescence data, new promoter comparisons, or new experimental conditions and still use the same basic structure: input data, normalization, analysis, and interpretation.

In practice, that means the Software page can become a stable record of how the team thinks about computational support. It can hold workflow diagrams, screenshots, summary plots, and notes on how dry-lab tools connect back to experimental design rather than sitting apart from it.
